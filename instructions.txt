PROJECT OVERVIEW
================

This is my personal remote-administration tool for two machines I own on my local
network. Both run Linux. I built it from scratch instead of using SSH because I
want practice writing custom protocols with raw sockets, and because I want
hostb to look closed to anyone scanning it.

There are two programs:
  - hosta.py: control center. I run this on my laptop. It shows a menu.
  - hostb.py: listener daemon. I run this on the other machine. It sniffs.

Both need sudo (raw sockets). Run as:
  sudo python3 hosta.py -a <hostb_ip>
  sudo python3 hostb.py -i <interface>


DESIGN RULES I CARE ABOUT
=========================

1. Raw sockets only. No TCP sockets via the normal API, no HTTP, no SSH.
2. Data travels in IP/TCP/ICMP HEADER FIELDS, never in packet payloads.
   I do this to practice repurposing header fields and to imagine a setting
   where the payload can't be read.
3. Identifier-field data is XOR-encrypted with a 16-bit pre-shared key
   (PRE_SHARED_KEY = 0xA5C3). It's barely encryption — that's deliberate
   for now. Sequence numbers are cleartext.
4. ICMP packets we send mimic real Linux `ping` exactly: type 8 echo request
   with an 8-byte zeroed timestamp slot + the 0x10..0x3F filler pattern (56
   total payload bytes). hostb ignores the payload but on the wire we look
   like normal ping traffic (98 bytes with ethernet header, not 42).
5. hostb should appear dark on the knock ports (use iptables DROP for true
   stealth: `iptables -A INPUT -p tcp -m multiport --dports 7000,8000,9000 -j DROP`).


CODE STRUCTURE (BOTH FILES)
===========================

Each file is organized like a small FSM. Reference style: my earlier
steg-encode.py and steg-decode.py tools in this same repo — use those as
the house style.

  - A `Context` dataclass holds args + runtime state + error_message + error_code.
  - Step functions take `ctx: Context` and modify it.
  - Errors: set ctx.error_message (and ctx.error_code if not 1) and call
    handle_error(ctx), which writes to stderr and sys.exit()s.
  - argparse with flag style (-a, -i), `prog=`, `metavar=`. The SystemExit
    catch normalizes argparse's exit-2 to exit-1.
  - Module orchestration lives in `if __name__ == "__main__":`, not a main()
    function, with banner headers and "\nXxx..." section markers.


CONNECTION: PORT KNOCKING
=========================

KNOCK_PORTS = (7000, 8000, 9000)
SOURCE_PORT = 54321 (hosta's source port on outgoing knocks + the port the ACK
                     comes back to)

Flow:
  1. hosta sends 3 raw TCP SYN packets to those three ports in order, ~100ms apart.
  2. hostb scapy AsyncSniffer (BPF filter for SYN-only on those three ports)
     tracks per-source-IP progress via KnockWatcher. 5s timeout if knocks are
     too spread out.
  3. On third success hostb sends a raw TCP ACK back: src port 9000, dst port
     54321, ACK flag only. (Wire-level forged — not a real TCP handshake.)
  4. hosta listens on a raw IPPROTO_TCP socket for that ACK from hostb's IP
     destined for SOURCE_PORT. Once seen, ctx.connected = True.

Quirk: hosta's kernel will fire a RST in response to the ACK because port
54321 isn't really bound. Harmless (our raw socket already captured the ACK).
To suppress: iptables -A OUTPUT -p tcp --tcp-flags RST RST --sport 54321 -j DROP.


COMMAND CHANNEL (POST-CONNECTION)
=================================

All commands and bulk data go over ICMP echo requests. Identifier carries the
data (encrypted), sequence carries packet ordering.

Command codes:
  CMD_DISCONNECT     = 1
  CMD_TRANSFER_FILE  = 2
  CMD_UNINSTALL      = 3
  CMD_RUN_PROGRAM    = 4
  ACK_READY          = 0xFFFE   (hostb signals "I'm ready for the data stream")

Two protocols for bulk data:

  ByteStreamWatcher (single-header, used by run_program):
    seq=1:        byte_count
    seq=2..N+1:   data, 2 bytes per packet

  FileReceiveWatcher (two-header, used by file transfer):
    seq=1:        filename_length
    seq=2:        file_size
    seq=3..M+2:   filename bytes, 2 per packet
    seq=M+3..end: file bytes,     2 per packet

Receivers key all packets by sequence number so out-of-order delivery is fine.
Each chunk is `(byte_high << 8) | byte_low`, encrypted via XOR with the
pre-shared key. The receiver truncates the reassembled buffer to the exact
declared length so odd-sized inputs don't carry padding.


MENU
====

hosta loops the menu after each action.

  1) Connect to hostb         — implemented (port knock + wait for raw ACK)
  2) Disconnect from hostb    — implemented (CMD_DISCONNECT). hostb drops back
                                to wait_for_session.
  3) Uninstall from hostb     — implemented (CMD_UNINSTALL). hostb wipes its
                                CWD (shutil.rmtree subdirs, os.remove files)
                                then sets stop_requested and exits. WARNING:
                                this wipes whatever directory hostb was started
                                from. Don't launch from the project source dir.
  4) Transfer file to hostb   — implemented. Prompts for a path, uses
                                os.path.basename so absolute paths don't leak,
                                sends filename + bytes. hostb saves as
                                received_<safe_basename>. Works for any file
                                type (binary or text). Cap 64KB.
  5) Transfer file FROM hostb — NOT implemented yet.
  6) Watch file on hostb      — NOT implemented yet.
  7) Watch directory on hostb — NOT implemented yet.
  8) Run program on hostb     — implemented. Prompts for a shell command,
                                ships it to hostb, hostb runs via
                                subprocess.run(shell=True, capture_output=True,
                                timeout=30s), ships stdout+stderr back, hosta
                                decodes UTF-8 (errors=replace for binary) and
                                prints between markers. Cap 64KB each way.


IMPLEMENTATION GOTCHAS WE'VE ALREADY HIT
========================================

- scapy AsyncSniffer.start() returns BEFORE the BPF socket is open. Packets
  arriving in the ~10-50ms window get silently dropped. Fix: after start(),
  wait on `sniffer.started` (Event in scapy >=2.5) or fall back to time.sleep(0.3).
  We do this in receive_file and run_program before sending ACK_READY, so
  hosta doesn't start firing data while hostb is still warming up.

- When hosta sends ICMP to hostb, hostb's kernel auto-replies with echo-reply
  (type 0). Our BPFs filter type 8 only, so they're ignored. Same when hostb
  sends ICMP to hosta — kernel reply is harmless noise.

- subprocess.run with shell=True and timeout: bash -c "cmd" exec-optimizes into
  the command directly (same PID), so the SIGKILL from timeout actually kills
  the target command. No zombies, no orphans, in practice.


THINGS LEFT OPEN / DECISIONS TO MAKE
====================================

- Encryption is still static XOR with a 16-bit key. Identical plaintext at
  the same sequence number always produces identical ciphertext. Considered
  upgrading to a per-packet keystream like SHA256(key || seq) → XOR.
- No reliability layer. Lost packets cause hangs until timeout. Could add
  per-packet acks + retransmits later, but it complicates the protocol.
- 64KB cap on file size and on command output comes from using a single
  16-bit identifier as the size header. Going larger needs a 2-packet size
  header (32-bit total).
- The sniffer-warm-up race could be eliminated entirely by keeping one
  sniffer running for the whole session and switching the watcher's mode
  (command vs file vs byte-stream) internally. Haven't refactored to that yet
  because the warm-up sleep works.
- Menu options 5/6/7 are unimplemented and I don't have firm designs yet.
  My intuition: "transfer file from hostb" mirrors option 4 in reverse;
  "watch file/directory" probably uses inotify on hostb and streams events
  back over the byte-stream protocol.
